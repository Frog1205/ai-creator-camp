# 教师教案 · Day 2：设计·开发

## 📌 教学目标
1. **认知**：理解"拆解 = 让AI能帮你"的工作方法
2. **技能**：能与AI协作产出可运行的HTML游戏原型
3. **情感**：经历"代码跑通"的成就感和"调bug"的挫败-恢复循环
4. **产出**：每组一份v0.1可运行游戏

## 🧰 课前准备
- [ ] 准备3–5个"通用代码模板"应急包（贪吃蛇/打砖块/接东西/记忆翻牌 各一份HTML）
- [ ] 检查所有机器能打开 .html 文件（默认浏览器设置）
- [ ] 教师本人**至少独立完成过一遍**整个流程（千万别第一次和学生一起做）

## ⏱ 课堂节奏

### 09:00–09:30 复盘 + 开工会（30'）
- 公布昨日"创意墙"投票TOP3 → 给他们鼓掌
- **重要！教今天的"圣经"**：
  > **一次只做一个功能，做完测试，再做下一个。**
- 在黑板写下来，留一整天。

### 09:30–10:30 任务1：拆功能清单（60'）
- 教师演示拆解1个不同的游戏（不要拆贪吃蛇——避免学生抄）
- 巡场时**强制**每组在白板上列出功能1/2/3/4/5
- ⚠️ 易错点：学生会列"画面好看"——这不是功能，是要求。引导**写动词**："玩家能用方向键移动"

### 10:30–12:00 任务2：第一版代码（90'）
**关键时刻**：让学生把代码**真的保存成 .html 并双击打开**。
- Windows记事本默认存 .txt，要选"所有文件"
- 引导建立**版本意识**：每跑通一版就 `game_v1.html`、`game_v2.html` 另存

### 14:00–15:30 任务3：调bug大作战（90'）
**这是今天教育价值最高的环节**。学生会卡，但不要直接帮他们解决——教他们**问AI**。

| 学生问你 | 你的回答（建议） |
|----------|-----------------|
| 老师我代码不动了 | "F12看一下控制台有没有红字，截图给AI。" |
| AI给的还是错的 | "把完整代码 + 期望结果 + 实际现象 一次性发给AI。" |
| 我不知道哪里错了 | "你先告诉我你**期望**它做什么，**实际**它在做什么。" |

🎯 **拐杖原则**：能让AI解决的，不要老师解决。**老师的价值是"教怎么问"，不是"给答案"**。

### 15:30–17:00 任务4：灵魂玩法（90'）
- 鼓励"小而美"——加一个音效胜过加5个功能
- 提醒：**不要碰核心代码**，只在外围加东西，否则容易把昨天跑通的版本搞坏

### 17:00–17:30 阶段演示 + 复盘（30'）
- 每组上来玩30秒自己的游戏给全班看
- 不评价"好不好"，只问"你今天最骄傲的一个改动是什么"

## 🚨 进度卡线 & 介入策略
今天结束时，每组**至少**应该达到：
- ✅ 一个可在浏览器打开的 .html
- ✅ 有玩家输入（键盘/鼠标）
- ✅ 有屏幕变化（动起来了）

**没达到的组**：
- 老师介入：给应急包模板（"这是别的组送你的初版"），让他们今晚回去**只改创意特色部分**
- **不要让任何一组带着"白板"过夜**——失去信心比落后进度更可怕

## 🚨 风险预案
| 风险 | 应对 |
|------|------|
| 整组代码崩了找不回旧版 | 应急模板兜底 + 强调以后每天最后存个稳定版 |
| 学生执着于一个炫酷功能 | 给硬性截止时间："15:30前必须有可玩版本，再炫的功能也得等" |
| AI生成的代码超长（500+行） | 引导"让AI删掉xxx功能给我精简版"——不会精简就用应急模板 |

## 📊 Day2 形成性评价点
- 是否真的在用"小步迭代"的方法
- 代码遇到问题时，是去问AI还是去问老师（前者得分高）
- 团队内部分工是否清晰（谁主控键盘、谁记录、谁测试）
