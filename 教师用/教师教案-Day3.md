# 教师教案 · Day 3：测试·迭代

## 📌 教学目标
1. **认知**：理解"用户反馈"是产品的终极标准，而非创作者的自我感觉
2. **技能**：能根据反馈做出**优先级判断**（什么先改、什么不改）
3. **情感**：体验"我的作品被别人玩"的真实喜悦与紧张
4. **产出**：v1.0完整版游戏 + 30秒Demo视频 + PPT骨架

## ⏱ 课堂节奏

### 09:00–10:00 互玩30分钟（60'）
**这是今天最重要的1小时**。
- 规则：**电脑留在原位，人轮转**。每个游戏玩2分钟，必须填反馈卡。
- 老师在场内**喊节奏**："2分钟到，下一组！"
- 玩完后留10分钟**让每组回到自己电脑读反馈卡** —— 不评论、只读。

🎯 **教学心法**：先让学生体会"被反馈砸中"的复杂情绪，再讲方法。

### 10:00–10:30 反馈分类工作坊（30'）
教学生用一个简单矩阵给反馈分类：

|  | 高频出现 | 只1人提 |
|---|---|---|
| **改起来简单** | ⭐ 立刻做 | 顺手做 |
| **改起来复杂** | 慎重评估 | ❌ 不做 |

每组花15分钟把反馈分进4格，决定今天做什么。

### 10:30–12:00 任务1：迭代+美化（90'）
**警惕陷阱**：很多学生会跳过功能改进直接美化（因为美化容易看到效果）。
- 强制顺序：先把"看不懂规则" / "操作不灵" 解决，**再**美化
- 教师巡场口令："功能层好了吗？没好不准换皮肤。"

### 14:00–15:00 任务2：关卡 / 难度（60'）
- 引导：关卡设计的本质是"难度曲线"
- 提问学生："第1关要让玩家感受到什么？第5关呢？"
- 这是**游戏设计思维**第一次真正登场——非常宝贵的瞬间，**不要赶**，宁可少改一个bug

### 15:00–16:00 任务3：录Demo视频（60'）
- 教师演示一次手机录屏 + 简单剪辑（系统相册自带的裁剪就够）
- 不要求精美——要求**真实**。一个抖动的手机录制比一个假的剪辑更有说服力。

### 16:00–17:00 任务4：PPT骨架（60'）
- 用《项目模板/路演PPT结构模板.md》
- 老师巡场只问一个问题："如果只能保留一张PPT，你保留哪一张？为什么？" → 帮学生找到自己的**核心故事**

### 17:00–17:30 排练（30'）
- 每组在原位讲一遍（不上台）
- 老师计时，**超过3:30的当场打断**，逼学生压缩

## 🎯 教学策略升级：今天开始"放手"
Day1–2 老师是引路人，Day3 开始**做减法**：
- 学生问"老师你看我这样行不行？" → "你觉得行不行？为什么？"
- 学生求安慰 → "明天演示完你就知道了。"
- 让他们**带着一点不确定上场**——这是PBL的精髓。

## 🚨 风险预案
| 风险 | 应对 |
|------|------|
| 某组反馈很差，士气崩了 | 单独谈话："反馈差≠作品差，可能是没让人get到。你们路演时要主动'教'玩家怎么玩。" |
| 某组开始攀比/嫉妒 | 公开讲："明天评奖有5个不同类别，每组都有机会赢。" |
| PPT做得花里胡哨 | 强调："3分钟，7页够了。一页一个意思，别让评委读字。" |

## 📊 Day3 形成性评价点
- 学生面对负面反馈的反应（防御 vs 好奇）
- 是否真的做了"取舍"——把不重要的反馈划掉
- 团队决策能力（争议时怎么决定）

## 🏠 教师当晚准备
- 准备评委名单（如有外部评委，提前发PPT结构告知）
- 准备评价量规（《教师用/评价量规.md》）打印多份
- 准备5个奖项的小奖品（不用贵，仪式感最重要）
